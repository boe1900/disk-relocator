# 健康检查：自动纠偏与一键回滚使用说明（V1）

日期：2026-03-06

## 目标

这份文档回答 4 个问题：

1. 系统自动纠偏是什么，何时触发。
2. `一键回滚` 是什么，何时该用。
3. 两者各自会改动什么，不会改动什么。
4. 异常时的推荐处理顺序。

## 先说结论（给普通用户）

1. 先点「健康自检（自动纠偏）」。
2. 程序会自动处理可证明无损的一致性问题。
3. 自检后仍异常，再使用 `一键回滚`。
4. 如果是“磁盘离线/只读”，先恢复磁盘状态再判断是否回滚。

## 功能定义

## 1) 自动纠偏（系统自动执行）

自动纠偏由后端 `reconcile_relocations(apply_safe_fixes=true)` 完成，目标是让“文件系统真实状态”和“数据库状态”保持一致。

当前仅覆盖无损修复：

1. 清理迁移临时残留目录 `*.tmp.<relocation_id>`。
2. 文件系统已健康时，将元数据状态纠正为 `HEALTHY`。
3. 文件系统已回滚时，将元数据状态纠正为 `ROLLED_BACK`。

不会做的事：

1. 不会复制大体积业务数据。
2. 不会重建丢失的数据目录内容。
3. 不会自动处理“数据来源不明确”的冲突场景。

## 2) 一键回滚（用户手动触发）

`一键回滚` 调用 `rollback_relocation({ force: true })`，目标是把迁移状态恢复回系统盘路径。

核心流程：

1. 删除源路径软链接（如存在）。
2. 优先用 `.bak` 恢复源路径；无 `.bak` 时尝试用 target 恢复。
3. 清理临时目录 `*.tmp.<relocation_id>`。
4. 状态写回 `ROLLED_BACK`，并记录健康快照和操作日志。

## 自动纠偏触发时机

1. 启动后的后台对账监控周期运行（自动）。
2. 用户点击「健康自检（自动纠偏）」时立即运行（前台触发）。

说明：页面不再提供单独“自动修复”按钮，避免用户理解负担。

## 典型场景 -> 用什么

## 场景 A：外接盘被拔出 / 未挂载

现象：

1. 常见 `HEALTH_DISK_OFFLINE`。
2. 状态通常 `degraded`。

推荐动作：

1. 先重新挂载磁盘。
2. 再点「健康自检（自动纠偏）」。
3. 若恢复 `healthy`，无需回滚。

## 场景 B：外接盘只读（权限问题）

现象：

1. 常见 `HEALTH_TARGET_READONLY` 或 `HEALTH_RW_PROBE_FAILED`。
2. 状态通常 `degraded`。

推荐动作：

1. 先修复磁盘可写权限。
2. 再运行自检。
3. 仍异常且影响使用时，再回滚。

## 场景 C：迁移中断后留下临时目录

现象：

1. 对账出现 `RECON_TEMP_PATH_RESIDUE`。

推荐动作：

1. 运行一次「健康自检（自动纠偏）」。
2. 系统会自动清理 temp 残留。

## 场景 D：元数据与文件状态不一致

现象：

1. 对账出现 `RECON_STATE_STALE_ACTIVE` 或 `RECON_STATE_STALE_ROLLED_BACK`。

推荐动作：

1. 运行自检，由系统自动纠偏。
2. 再次自检确认。

## 场景 E：断链/目标目录丢失/源路径异常

现象：

1. 例如 `HEALTH_TARGET_MISSING`、`HEALTH_SYMLINK_MISSING`。
2. 通常不在自动纠偏覆盖范围。

推荐动作：

1. 先判断外接盘是否在线。
2. 无法快速恢复时，执行 `一键回滚` 恢复系统盘路径。

## 成功与失败边界（重点）

## 自动纠偏边界

1. 仅覆盖无损小动作，风险低。
2. 不能解决“数据真实内容丢失”或“来源冲突”问题。

## 回滚边界

回滚成功率最高前提：

1. 目标应用已退出。
2. `.bak` 或 target 至少有一个可用恢复来源。

常见失败：

1. `ROLLBACK_RELOCATION_NOT_FOUND`：记录不存在。
2. `ROLLBACK_REMOVE_SYMLINK_FAILED`：删除软链失败（占用/权限）。
3. `ROLLBACK_RESTORE_BACKUP_FAILED`：既无 `.bak` 也无法从 target 恢复。
4. `ROLLBACK_CLEANUP_TEMP_FAILED`：临时目录清理失败。

特别提示：

1. 若迁移时已清理 `.bak`，且外接盘离线或损坏，回滚可能缺少恢复来源而失败。
2. 因此“磁盘离线”场景应优先恢复磁盘在线，再做回滚。

## 推荐操作顺序（SOP）

1. 关闭目标应用进程（例如微信）。
2. 点击「健康自检（自动纠偏）」。
3. 查看结果；若仍异常且影响使用，执行 `一键回滚`。
4. 回滚后再次自检确认。

备注：回滚成功后，记录会进入 `ROLLED_BACK`，通常不再显示在健康检查列表中，这是预期行为。

## 如何定位问题

1. 记录 `relocation_id` 与时间。
2. 在「操作记录」中查看失败详情（关键步骤 + 错误码 + 错误信息）。
3. 重点看 `health` / `rollback` 阶段的错误码。

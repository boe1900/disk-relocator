# 可行性与边界评估

## 1. 问题定义

目标用户：
- 256GB 版本 Mac mini / MacBook Air 用户
- 微信、Telegram、Docker、开发工具重度用户

核心痛点：
- `~/Library` 下数据增长快，极易吃满内置盘
- 现有工具大多只能搬 `.app`，难管理数据目录迁移
- 软链接能做，但缺少统一台账、健康检测和一键还原

## 2. 现有工具与缺口

| 工具 | 优势 | 主要局限 |
|---|---|---|
| AppCleaner & Uninstaller | 能找关联文件，也能搬应用包 | 对 Data Container 的软链接生命周期管理较弱 |
| SymbolicLinker | 右键即可建软链接，门槛低 | 没有集中管理界面，后期不可维护 |
| Hazel | 自动化能力强 | 规则配置复杂，不适合多数普通用户 |

结论：
- 市场有相邻能力，但没有“专门做应用数据迁移 + 可视化管理 + 安全回滚”的现代工具。

## 3. 技术可行性（Tauri + Rust）

可行性结论：高，可落地，但有边界。

容易实现的部分：
- 扫描候选路径
- 统计体积
- 复制/迁移数据
- 创建软链接
- 记录迁移元数据

需要重点设计的部分：
- macOS 权限（`Full Disk Access`）
- `~/Library/Containers` 下沙盒应用兼容性
- 外接盘离线/只读/掉线场景处理

## 4. 不可回避的产品边界

1. 不是所有应用都支持迁移，必须公开兼容性分级。
2. MAS/沙盒应用通常只能标记为 `Experimental` 或 `Blocked`。
3. 运行中迁移必须禁止，否则有数据损坏风险。
4. 外接盘离线必须可感知、可告警、可恢复。

## 5. V1 Go/No-Go 标准

满足以下条件才可发布：

1. 至少 3 个高频应用可稳定完成“迁移-使用-回滚”闭环。
2. 任一步骤中断后可一键恢复一致状态。
3. 外接盘离线能在用户打开受影响应用前给出明确预警。
4. 重启后元数据状态与文件系统实际状态一致。

以下任一出现即不可发布：

1. 中断场景下无法保证数据完整性。
2. 回滚后仍会残留断链或半迁移状态。
3. 支持列表无法做到“可验证、可追踪、可复现”。
